# AES67 PC 软件栈 — 开发计划

> 本计划以**接手现有代码、在既有骨架上推进到 `方案设计.md` 最终形态**的视角编写。
> 说明"从现有代码的哪里接着做、分几个阶段、每阶段交付什么、怎么验证"。
> 配套：`方案设计.md`（目标形态）、`CODE_REVIEW.md`（现状与 bug 清单，含精确行号）。

---

## 1. 现有代码骨架盘点（起点）

接手时仓库已有的可复用资产：

| 目录 / 文件 | 现有内容 | 在本计划中的角色 |
|---|---|---|
| `engine/aes67_engine.*` | 引擎主控 + 状态机 + 启停 | ★ 常驻引擎地基，直接在其上扩展 |
| `engine/network_thread / network_receiver` | RTP 收发线程 | 复用，修合规问题后即可用 |
| `engine/sap_announcer.*` | SAP/SDP 通告 | 复用，需修地址 bug |
| `engine/jitter_buffer.h / ring_buffer.h` | 抖动 / 环形缓冲 | 复用，需修竞态 |
| `engine/ptp_clock.* / ptp_thread.*` | PTP 从时钟 | 复用，跨机阶段再深化 |
| `engine/wasapi_device.*` | WASAPI 采集 / 渲染 | 音源输入的**第一实现**（先于驱动共享内存） |
| `engine/pipe_server.*` | 命名管道 server | ★ 面板↔引擎接口地基，骨架已成型 |
| `panel/`（Qt + `pipe_client`） | Qt 面板 + 管道客户端 | ★ 面板地基，路由矩阵在此扩展 |
| `driver/`（Scream/MSVAD 派生） | WDM 驱动 + IOCTL + 共享内存桩 | 制造声卡端点；共享内存通路待打通 + 清残留 |
| `asio/asio_minimal.cpp` | 自包含 ASIO DLL | 可选客户端，非主线 |
| `build.py` | engine / asio / panel / driver 构建脚本 | 逐阶段补全 |

**接手判断**：引擎与 RTP 收发是能自洽运行的实体，命名管道 + Qt 面板刚起步、方向正确。因此计划**不推倒重来**，而是沿现有骨架：先让"引擎 + 面板"这套可用且合规 → 再往上叠路由 / 混音 / DSP → 最后打通内核收流与共享内存。

## 2. 协作方式

| 项 | 约定 |
|---|---|
| 代码归属 | `dwygg/V_AES67`，直接在此仓库开发 |
| 协作模式 | **feature 分支 + PR**：每阶段推 `feature/Pxx-xxx` 分支，Windows 拉取编译验证，通过后合入 `main` |
| 开发环境 | macOS 侧写代码（用户态 C++/Qt 全可写），**无法本机编译** |
| 验证环境 | Windows + VS2022 + Qt6（后期加 WDK）；每次 push 都是**能独立编译验证的完整单元** |
| 验收手段 | 首选本机自发自收 / 第二台机 + **Wireshark 抓包**验合规；进阶用真实 AES67 设备（顶力）订阅验互通 |

**协作循环**：① mac 写代码推 `feature/Pxx` → ② Windows checkout 该分支编译 → ③ 反馈报错 / 抓包 / 现象 → ④ 改后再推，循环到完成判据全绿 → ⑤ 合 main 打 tag，进下一阶段。

> ⚠️ 铁律：**每次 push 必须能编过**。macOS 侧无法编译，一次推入多个未验证改动会让往返成本爆炸。宁可拆小、多推几次。

## 3. 阶段总览

```
用户态（可在 mac 写，Windows 编译验证）        │ 内核（需 WDK，最硬，后置）
──────────────────────────────────────────┼──────────────────────
P0 环境自检：现有代码能编过、能跑           │
P1 修 M9 死锁 + pipe 生命周期（面板可控引擎）│
P2 修 SAP 地址 + 锁死音频格式（流可被订阅）⭐│
P3 路由契约(JSON) + 引擎读表               │
P4 多源汇聚（常驻引擎混音总线）             │
P5 Qt 面板接路由矩阵（可视化 + 下发）        │
P6 DSP 挂载点（增益 / 静音 / EQ 骨架）      │
──────────────────────────────────────    │ P7 内核收流 → 虚拟麦克风输入端点
                                          │ P8 打通 IOCTL + 共享内存（终局）
```

**首阶段锁定 P1 + P2**：先让"引擎 + 面板"这套现有骨架稳定可控、发出的流能被真设备订阅到——风险最低、成果最直观，是后续一切的地基。

## 4. 阶段详情

### P0 — 环境自检（清地基，非开发）

先确认 Windows 侧能把**现有代码**编过、跑起来，排除"环境问题"与"改动问题"混淆。

| 步骤 | 动作 | 判据 |
|---|---|---|
| 0.1 | `python build.py engine` | 生成 engine，能启动、监听 `\\.\pipe\AES67Engine` |
| 0.2 | `python build.py asio` | 生成 `AES67_ASIO.dll`（可选客户端，仅确认能编） |
| 0.3 | 装 Qt6 + CMake 编 `panel/` | 生成面板 exe，能启动 |
| 0.4 | 记录 VS / Qt / WDK 版本 | 据此对齐 `build.py`（当前 `build_panel()` 写死了 Qt 路径与生成器名，需改成可配置） |

**产出**：一份"Windows 环境 & 首次编译结果"反馈（哪步过、哪步报什么错）。

---

### 🟢 P1 — 修 M9 死锁 + pipe 生命周期　`feature/P1-m9-fix`

**现有代码起点**：`engine/aes67_engine.cpp` 的 pipe 挂载与 handler、`engine/pipe_server.*`、`panel/mainwindow.cpp` 的设置下发。（bug 精确行号见 `CODE_REVIEW.md §2`。）

| # | 现状问题 | 在哪改 | 改法 |
|---|---|---|---|
| M9-1 | STOP 命令在 pipe 线程里调 `Stop()`，`Stop()` 又去 join pipe 线程 → 自锁卡 3s | `aes67_engine.cpp:119` handler + `:223` | handler 不直接调 `Stop()`；改为置原子标志，交主循环执行；或 `Stop()` 内判断"在 pipe 线程则跳过停 pipe" |
| M9-2 | pipe 绑在音频 `Start()`（`:158`）→ 引擎未 Start 时面板永远连不上 | `aes67_engine.cpp` | `m_pipeServer.Start()` 移到 `Initialize()`；`Stop()` 删掉 `:223` 停 pipe；pipe 仅在 `Shutdown()` 停 |
| M9-3 | SET 只改 `m_netConfig` 内存值、不重建 socket → 改地址不生效 | `aes67_engine.cpp` SET handler | SET 后安全重启网络线程，或回报"需重启生效" |
| M9-4 | 面板 Apply 漏发 source port（`m_sourcePort` 从未被读） | `panel/mainwindow.cpp` | 面板补发；引擎 SET 解析使用 |

**完成判据**：① 引擎一启动（未 Start 音频）面板即可连上显示状态；② 面板 START/STOP 正确启停且不卡顿；③ 改地址 → Apply → 抓包确认发流目标随之改变。

---

### 🟢 P2 — 修 SAP 地址 + 锁死音频格式（让流可被订阅）⭐　`feature/P2-sap-format`

这是"能否与顶力 / 其它 AES67 设备互通"的头号拦路虎。**现有代码起点**：`engine/sap_announcer.*`、`engine/aes67_engine.cpp:178` 的 SAP 调用、`engine/audio_config.h`。

| # | 现状问题 | 在哪改 | 改法 |
|---|---|---|---|
| SAP-1 | `aes67_engine.cpp:178` 调 SAP `Start()` 只传 3 参，`mcastAddr` 走默认 `239.255.255.255`（SAP 组地址），SDP `c=IN IP4` 因此错通告成 SAP 地址 | `aes67_engine.cpp:178` | 显式传入真实 RTP 组播目标地址 |
| FMT-1 | `audio_config.h` 默认虽 48k/24bit/2ch，但 `periodUs=100000`（=100ms，注释误写 10ms），AES67 需 **ptime=1ms（48 样本）** | `audio_config.h` | 锁死 L24 / 48kHz / 2ch / ptime=1ms |
| SDP-1 | SDP 需符合 SMPTE ST 2110-30：`RTP/AVP 97`、`rtpmap:97 L24/48000/2`、`ptime:1`、`ts-refclk` | `sap_announcer.*` | 补齐 SDP 字段；通道命名可读 |

**完成判据**：① Wireshark 抓 SAP：`c=` 是真实 RTP 组播地址，`rtpmap/ptime` 符合 L24/48k/1ms；② 抓 RTP：PT=97、每包 48 样本、时间戳步进 48、1000 pkt/s；③（有设备时）顶力 / 第二台能**发现并订阅**到流、听到声音。

> **P2 末尾 = 第一个关键验收点（MS-2）**：发出的流被真设备订阅到，对外契约立住。

---

### 🔵 P3 — 路由契约（JSON）+ 引擎读表　`feature/P3-routing-schema`

**现有代码起点**：`engine/aes67_engine.cpp`（引擎主控读配置）、`engine/pipe_server.*`（新增命令）。当前引擎只有单路固定收发，尚无路由概念——本阶段引入。

- 定义 `routing.json` schema：`源通道 → 目标流/通道 + 增益 + 静音 + 流描述`。
- 引擎启动读表；pipe server 新增 `GET_ROUTING` / `SET_ROUTING` 命令。
- 此阶段**手改 JSON 即可测通路由逻辑，不依赖界面**。

**完成判据**：改 JSON → 重启 → 路由 / 流数量 / 增益按表生效（抓包确认多流）。

---

### 🔵 P4 — 多源汇聚（常驻引擎混音总线）　`feature/P4-mixer-bus`

**现有代码起点**：`engine/wasapi_device.*`（先用 WASAPI 回环作多路音源）、引擎主控新增混音总线模块。

- 引擎内建混音总线：多路输入（阶段用 WASAPI 回环，后续换驱动共享内存）按 P3 路由表混合 / 分发到目标流。
- 总线是后续 DSP 的挂载点。
- 顺带清理 `jitter_buffer.h` 的半包读竞态（memcpy 后再 release 写 valid），保证多流下缓冲正确。

**完成判据**：两路以上音源能按路由表混合 / 分发到不同 AES67 流。

---

### 🔵 P5 — Qt 面板接路由矩阵　`feature/P5-panel-matrix`

**现有代码起点**：`panel/mainwindow.*`（新增 tab）、`panel/pipe_client.*`（复用 P3 契约）。

- 面板新增"路由矩阵"tab：源 × 目标网格、增益条、静音钮。
- 通过命名管道复用 P3 的 `GET_ROUTING`/`SET_ROUTING`；面板只展示 / 编辑，逻辑在引擎。

**完成判据**：面板改路由 → 引擎实时生效 → 抓包可见。

---

### 🔵 P6 — DSP 挂载点（增益 / 静音 / EQ 骨架）　`feature/P6-dsp`

**现有代码起点**：P4 的混音总线模块 + `panel/`。

- 混音总线上留 DSP 处理链插槽：先做增益 / 静音 / 简单 EQ，面板可调参。

**完成判据**：面板调 DSP → 输出音频 / 抓包频谱可见变化。

---

### 🟠 P7 — 内核：收流 → 虚拟麦克风输入端点（WDK，最硬）　`feature/P7-capture-endpoint`

**现有代码起点**：`driver/`（Scream/MSVAD 派生，当前只有 render）。

- 现有驱动只有 render（输出），缺 capture（输入）。要让"网络来的 AES67 音频"成为系统录音设备，需 WDK 新增 capture pin。
- 需在 Windows 侧深度参与（测试签名 / 测试模式）。

**完成判据**：系统"录音设备"出现虚拟麦克风，DAW 能录到网络流。

---

### 🟠 P8 — 打通 IOCTL + 共享内存（终局）　`feature/P8-ioctl-shm`

**现有代码起点**：`driver/adapter.cpp`（IOCTL handler + `g_SharedBuffer`）、`ioctl_test.cpp`、引擎侧新增共享内存桥。

- 实现方案设计 §2.2 的"驱动 memcpy 音频到共享内存、引擎经 IOCTL 读写"，替换 WASAPI 回环，走向"薄内核 + 厚引擎"终局。
- 先修 IOCTL 码不匹配（`ioctl_test.cpp` 的 `0x22E0000` vs 驱动实际 `0x222003`）与共享内存死缓冲（`adapter.cpp` 只分配清零不写音频）。
- 同步清理驱动内 Scream 残留（WSK 组播 / ivshmem / 换皮标识）。

**完成判据**：引擎经 IOCTL + 共享内存与驱动交换音频，WASAPI 回环可摘除。

## 5. 里程碑验收点

| 里程碑 | 阶段 | 验收标志 |
|---|---|---|
| MS-1 可控 | P1 | 面板稳定控引擎，不死锁 |
| **MS-2 可被订阅 ⭐** | P2 | 发出的流被真 AES67 设备 / 第二机订阅到（对外契约立住） |
| MS-3 可路由 | P3+P4 | 多源按表混合 / 分发到多流 |
| MS-4 可视化 | P5+P6 | 面板路由矩阵 + DSP 可用 |
| MS-5 可录入 | P7 | 系统出现虚拟麦克风，能录网络流 |
| MS-6 终局 | P8 | IOCTL + 共享内存打通，摘除 WASAPI 回环 |

## 6. 开始前需你拍板的事项

1. **P0 自检结果**：现有 `build.py engine/asio` + `panel/` 在你 Windows 上能编过吗？把结果反馈我，再开 P1。
2. **分支命名**：用 `feature/P1-m9-fix` 这种，行吗？
3. **P2 验收条件**：能凑到第二台机 / 顶力设备吗？没有的话以 Wireshark 抓包合规为准。
4. **驱动残留处理时机**：Scream 残留（WSK 组播 / ivshmem / 死缓冲）确认留到 P8 一并清理，还是希望更早清掉以免误导？

确认后（尤其第 1 项环境自检），即可从 **P1 分支**正式开写。
