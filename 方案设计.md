## 1. 产品定义

### 1.1 一句话

把整台 Windows 变成一块**常驻的、能汇聚全系统声音的专业 AES67 网络声卡**：应用产生的音频进入系统 → 引擎按路由表混音、经 DSP 处理 → 以标准 AES67 流发送到网络；同时能订阅网络上其它 AES67 设备的音频流，注入成系统录音设备。整套能力通过 Qt 面板配置，并与第三方 AES67 设备（如顶力 USB↔AES67 转换器）在同一网络自由互操作。

### 1.2 三层软件栈

专业 AES67 / Dante 生态里，PC 端不是"一个驱动"就完事，而是**三层解耦、又能联合使用**的软件栈：

| 层 | 作用 | 类比（Dante 世界） |
|---|---|---|
| **① 虚拟声卡驱动** | 系统里出现一块声卡（输出 + 输入端点），把应用声音收进来、把网络流放出去 | Dante Virtual Soundcard |
| **② 声道映射 / 路由** | 决定"哪路音源 → 网络哪个通道"、订阅哪些收流，可视化矩阵配置 | Dante Controller |
| **③ 音效 DSP** | 混音总线上的增益 / 静音 / EQ / 限幅等处理 | 各家插件 / 调音台软件 |

三层各自独立（可以只用驱动、不开 DSP），也能联合（路由软件读驱动的通道、DSP 挂在路由后的总线上）。支撑这三层的地基，是一个**能同时看到所有音源与所有网络通道的常驻音频引擎**。

### 1.3 目标用户与场景

- 把一台 Windows 主机当作录音棚 / 演播室里的**常驻网络音频节点**：所有应用声音统一走网、统一时钟。
- 与专业 AES67 硬件（调音台、USB↔AES67 转换器、网络音箱）在同一张 IP 网络里互相收发。
- 需要"整机声音路由 + 总线 DSP"的桌面音频工作流。

## 2. 系统架构

```
┌──────────────────────────────────────────────────────────┐
│                   Qt 配置面板 (用户态)                     │
│         状态监控 · 设置 · 声道路由矩阵 · DSP 调参          │
└───────────────────────────┬──────────────────────────────┘
                            │ 命名管道 \\.\pipe\AES67Engine
                            │ (文本命令 + 路由/DSP JSON)
┌───────────────────────────▼──────────────────────────────┐
│              AES67 音频引擎 (用户态 · 常驻服务)            │
│  ┌────────┐┌────────┐┌──────────┐┌──────┐┌───────────┐   │
│  │混音总线││路由表  ││jitter buf││ ASRC ││ DSP 处理链│   │
│  └────────┘└────────┘└──────────┘└──────┘└───────────┘   │
│  ┌────────────┐┌───────────┐┌─────────────┐             │
│  │RTP 收发线程││PTPv2 从时钟││SAP/SDP 通告 │             │
│  └────────────┘└───────────┘└─────────────┘             │
└──────┬───────────────────────────────────┬──────────────┘
       │ 音源 (发送侧) / 注入 (接收侧)        │ RTP / UDP
┌──────▼──────────┐              网络 (PTP 时钟同步) ──► 第三方 AES67 设备
│  虚拟声卡驱动    │                            ▲
│  (WDM/PortCls)  │◄───────────────────────────┘ 收流 → 虚拟麦克风端点
│  Speaker + Mic  │
└─────────────────┘
```

### 2.1 核心工程原则

- **常驻引擎是地基**：所有智能（路由 / 混音 / RTP / PTP / ASRC / SAP / DSP）集中在用户态引擎；进程崩了可重启、可脱离驱动单测、可跨平台复用逻辑。
- **面板只做展示 / 控制**：通过命名管道与引擎通信，本身不碰音频。面板崩溃不影响音频流。
- **内核薄**：驱动只负责"制造声卡端点 + 在驱动与引擎之间搬字节"，零业务逻辑，绝不在内核跑网络协议栈。协议状态（RTP 序号 / PTP 时钟 / ASRC 相位）全在用户态，内核有 bug 最多丢音频而不蓝屏。
- **音频线程实时**：无锁、无系统调用、无动态内存（一次性预分配）；跨线程共享变量全用 `std::atomic` + Memory Barrier。

### 2.2 数据通路

**发送方向**：应用输出 → 系统声卡端点 → 驱动搬到引擎输入总线 → 引擎按路由表混音 + DSP → RTP 封包 → 组播发送到网络。

**接收方向**：网络 RTP 流 → 引擎收包 → jitter buffer 平滑 → 按路由注入到驱动的虚拟麦克风端点 → 系统录音设备 → DAW / 应用录入。

**引擎与驱动之间的音频搬运**采用**共享内存 + IOCTL**：驱动把 PCM `memcpy` 进一块非分页共享内存，引擎经 `DeviceIoControl` 获取地址与读写指针后直接映射读写，不经过 Windows 音频混音栈，延迟低、路径短。读写指针用 `volatile` + Memory Barrier 实现无锁生产者/消费者。

## 3. 模块分解

### 3.1 虚拟声卡驱动（内核态，WDM/PortCls）

- 注册为系统声卡，暴露 **Speaker（输出）+ Microphone Array（输入）** 两组端点。
- 输出端点：接收系统混音数据，搬入共享内存供引擎读取（发送方向的音源）。
- 输入端点：从共享内存读取引擎解码后的网络音频，作为录音源（接收方向的注入）。
- 自定义 IOCTL：`GET_SHARED_BUFFER`（返回共享内存地址）、`GET_POSITION`（读写指针）、`SET_FORMAT`（协商采样格式）。
- 驱动内**不做任何网络 I/O、不解 RTP、不算时钟**——只 `memcpy` + 维护指针。

### 3.2 AES67 音频引擎（用户态，常驻服务）

| 子模块 | 职责 |
|---|---|
| **主控 / 状态机** | 进程生命周期、启停、线程编排 |
| **共享内存桥** | 经 IOCTL 与驱动交换音频（发送取、接收注入） |
| **混音总线** | 汇聚多路输入，按路由表分发；DSP 挂载点 |
| **路由表** | `源通道 → 目标流/通道 + 增益 + 静音`，可热加载 |
| **RTP 发送线程** | L24 封包，组播发送，独立于音频线程 |
| **RTP 接收线程** | `recvfrom` + jitter buffer + 丢包静音填充 |
| **PTPv2 从时钟** | Sync/Follow_Up/Delay_Req/Delay_Resp 四步 + 卡尔曼滤波估 offset/drift |
| **ASRC 采样率转换** | 用本地时钟与主时钟之差微调采样率，防跨机爆音 |
| **SAP/SDP 通告** | 周期性组播 SDP 会话描述，供接收方发现订阅 |
| **DSP 处理链** | 混音总线上的增益 / 静音 / EQ / 限幅插槽 |
| **控制接口** | 命名管道 server，收发面板的控制命令与路由/DSP 配置 |

### 3.3 Qt 配置面板（用户态，独立进程）

- **状态监控**：引擎运行状态、PTP 同步状态、收发包统计、时钟偏差。
- **设置**：目标 / 源组播地址、端口、音频格式、网卡选择。
- **声道路由矩阵**：源 × 目标网格，增益条、静音钮，可视化连线。
- **DSP 调参**：每条总线的 EQ / 增益 / 限幅。
- 全部经命名管道与引擎通信，面板只是"引擎状态的一个视图 + 命令下发器"。

### 3.4 ASIO DLL（可选客户端）

为需要低延迟直连的 DAW 提供 ASIO 接口，作为引擎的一个**客户端**接入（而非独立的第二套协议栈）。属可选组件，不在核心链路上。

## 5. AES67 标准合规与互通

**互通的根本保证是严格遵循 AES67 标准，而非去适配任何厂商的私有软件。** 做到标准合规，本产品与第三方 AES67 设备天然可在同一网络互操作。

### 5.1 流格式（SMPTE ST 2110-30 / AES67）

| 项 | 规格 |
|---|---|
| 编码 | L24（24-bit 线性 PCM，大端） |
| 采样率 | 48 kHz（主档），可扩展 96 kHz |
| 声道 | 2ch（可扩展多通道） |
| 打包时间 ptime | 1 ms（48 样本/包，1000 pkt/s） |
| RTP payload type | 动态 97 |
| 传输 | UDP 组播，RTP over IP |

### 5.2 会话描述与发现（SDP / SAP）

- SDP 必须正确通告：`m=audio <port> RTP/AVP 97`、`a=rtpmap:97 L24/48000/2`、`a=ptime:1`、`c=IN IP4 <真实 RTP 组播地址>`、`a=ts-refclk`（PTP 时钟域）。
- 发现机制以 **SAP** 为主（周期组播 SDP），并预留对齐 **Ravenna / mDNS**、**NMOS IS-04/05** 的能力——需与对端环境所用机制一致，接收方才能发现本设备。
- 通道命名（SDP `i=`、通道 label）保持人类可读，便于第三方路由软件把本设备通道排进矩阵。

### 5.3 时钟

- 作为 PTPv2 从时钟，跟随网络主时钟（Grandmaster）。
- 本地采样时钟与主时钟的漂移由 ASRC 补偿，保证长时间运行不累积爆音。

## 6. 关键工程问题与应对

| 问题 | 机制 | 应对 |
|---|---|---|
| 爆音 · 时钟漂移 | 本地晶振与主时钟差几十 ppm，累积误差 | ASRC 动态微调采样率，调幅 ≤ 0.1% |
| 爆音 · 网络抖动 | 网络丢包 / 重排 / 延迟尖峰 | jitter buffer 动态窗口 2–6 包；丢包静音填充不中断流 |
| 时钟精度 | Windows 软件时间戳约 1 ms，PTP 常要求 < 100 μs | QPC + PreciseSystemTime 双时钟 + 卡尔曼滤波 |
| 驱动稳定性 | buffer 位置 / IOCTL 超时 | 驱动只 memcpy、协议全在用户态；位置偏移 > 1 ms 时重置 |
| 驱动签名 | WHQL 认证成本 | 开发期用测试签名（testsigning on），发布前申请正式签名 |

## 7. 代码质量准则

- **内核驱动**：C，不用异常 / 标准库 / 动态内存（一次性 alloc at init）；回调只 memcpy，不解包。
- **用户态引擎**：C++17，RAII，智能指针，`std::vector`/`std::array` 代替裸指针。
- **音频线程**：无锁、无系统调用、无未预分配的内存操作、不写详细日志。
- **跨线程共享**：全用 `std::atomic`；共享内存用 Memory Barrier。
- **可测性**：每模块独立可测——引擎脱离驱动测、驱动脱离网络测。
- **日志**：分级输出；实时路径只置标志，日志在非实时线程落地。
