# AES67 虚拟声卡 — 软件开发方案

> 版本 V1.0 | 状态：方案评审
> 目标：开发一款支持 WDM + ASIO 双模式的 AES67 虚拟声卡，跨 Windows/macOS

---

## 1. 产品定义

### 1.1 功能描述

安装后在系统音频设备列表注册为声卡（WDM），支持 ASIO 低延迟通道。音频数据实时编码为 AES67 流（RTP/UDP + PTPv2 时钟同步）发送至网络；同时从网络接收 AES67 流解码为系统可录音源。

### 1.2 系统架构

```
Windows 应用 (浏览器/播放器/DAW)
     │ WDM              │ ASIO
     ▼                  ▼
┌─────────┐      ┌─────────────┐
│ WDM 驱动 │      │  ASIO DLL   │  ← 用户态 DLL
│ (内核态)  │◄─────│  (IOCTL直通) │
└────┬────┘      └──────┬──────┘
     │                  │
     │  共享内存 / IOCTL  │
     └────────┬─────────┘
              ▼
┌─────────────────────────┐
│     AES67 音频引擎       │  ← 用户态服务进程 (C++17)
│  ┌────────┐ ┌─────────┐ │
│  │RTP收发 │ │PTPv2时钟│ │
│  └────────┘ └─────────┘ │
│  ┌────────┐ ┌─────────┐ │
│  │AES67封装│ │ASRC校准 │ │
│  └────────┘ └─────────┘ │
└────────────┬────────────┘
             │ RTP/UDP
        网络 (PTP 时钟同步)
             │
┌─────────────────────────┐
│    Qt 6.8 配置 GUI       │  ← 用户态桌面程序
│  - 设备发现/通道映射      │
│  - 采样率/延迟配置        │
│  - 状态监控              │
└─────────────────────────┘

备注：DSP 效果处理由第三方提供，不在本方案范围。
```

## 2. 技术选型

| # | 模块 | 语言 | 工具/库 | 选型理由 |
|------|------|------|------|------|
| 1 | WDM 内核驱动 | C | WDK 10 + MSVAD 模板 | MSVAD 已有设备注册、buffer管理、IOCTL骨架。手写无意义 |
| 2 | ASIO DLL | C | Steinberg ASIO SDK 2.3 | 官方维护，免费分发 |
| 3 | AES67 音频引擎 | C++17 | 纯标准库 + socket API | 无需框架。RTP/UDP 收发轻量；SAP/SDP 固定模板 |
| 4 | PTPv2 时钟 | C++17 | 自写 | 软件时间戳 + 统计滤波。不依赖硬件支持 |
| 5 | ASRC 采样率转换 | C | `libsamplerate` | 行业标准，解决时钟漂移导致的爆音 |
| 6 | 配置 GUI | C++/Qt 6.8 | Qt Widgets | 跨平台、设置面板、状态监控 |
| 7 | 编译工具 | VS 2022 + WDK 10 | Windows 10/11 x64 |

**不用**：gstreamer/FFmpeg（太重，AES67 不需要编解码）、Qt（底层协议栈不需要框架）

## 4. 关键问题与应对

### 4.1 爆音

| 原因 | 机制 | 应对方案 |
|------|------|------|
| 时钟漂移 | PTP 主时钟 48kHz，本地晶振 48001Hz，累积 1ms/s 误差 | ASRC 动态微调采样率，调幅 ≤ 0.1% 听不见 |
| 网络抖动 | WiFi 丢包/重排/延迟尖峰 | jitter buffer 多缓冲 2-6 个包周期，宁可延迟高 5ms 不爆音 |
| Windows 调度抢断 | DPC/中断 >3ms 导致 WDM 回调超时 | MMCSS 注册音频线程优先级；驱动回调只做 memcpy，不解包 |

### 4.2 掉驱动

| 原因 | 应对方案 |
|------|------|
| buffer 位置不准 | `KeQueryPerformanceCounter` 校验播放位置，偏移差 >1ms 重置 |
| IOCTL 超时 | 驱动回调只 memcpy；协议处理全部在用户态 |
| 休眠唤醒 | WDM `EvtDeviceD0Entry/D0Exit` 处理正确；唤醒后重置 buffer 状态 |

### 4.3 时钟对时精度不足

Windows 软件时间戳精度 ~1ms。PTPv2 通常要求 <100μs。解决方案：
- 使用 `QueryPerformanceCounter` + `GetSystemTimePreciseAsFileTime` 双重时钟
- 卡尔曼滤波平滑 offset 波动
- 对音频目标：不需要硬件级 PTP 精度——ASRC 补偿 100μs 漂移完全足够

### 4.4 签名认证

WHQL 签名 $400/年。开发阶段用测试签名（`testsigning on`），发布前再申请正式证书。

### 4.5 跨平台同步

共享代码：音频引擎 100% 跨平台（C++17 + 标准库 + socket API）。平台相关代码约 20%——WDM/ASIO vs CoreAudio vs ALSA 的注册和 buffer 管理。
