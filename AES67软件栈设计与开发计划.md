## 1. 产品定义

### 1.1 一句话

把整台 Windows 变成一块**常驻的、能汇聚全系统声音的专业 AES67 声卡**，并提供**声道路由**与 **DSP** 能力，能与第三方 AES67 设备（如顶力 USB↔AES67 转换器）在同一网络互操作。

### 1.2 为什么是「形态 B」而不是「单 DAW 直连」

专业 AES67 / Dante 生态里，PC 端从来不是「一个驱动」就完事，而是**三层解耦、又能联合**的软件栈：

| 层 | 作用 | 类比（Dante 世界） |
|---|---|---|
| **① 虚拟声卡驱动** | 系统里出现一块声卡，把应用声音收进来 | Dante Virtual Soundcard |
| **② 声道映射 / 路由** | 决定「哪路音源 → 网络哪个通道」、订阅哪些收流 | Dante Controller |
| **③ 音效 DSP** | 混音总线上的 EQ / 限幅 / 增益 | 各家插件 / 调音台软件 |

这三层能成立，**前提是有一个「能同时看到所有音源和所有网络通道」的常驻引擎**——这就是形态 B。对比另一条被否定的路线：

| | A. 单 DAW 直连（早期 MVP，已否定为主线） | **B. 整机 AES67 软件栈（本项目目标）** |
|---|---|---|
| 音源范围 | 只有挂在 ASIO 上的那个 DAW | 整机所有应用（DAW / 浏览器 / 系统音） |
| 数据通路 | ASIO DLL 内直接发 RTP，不经过驱动 | 应用 → 驱动 / WASAPI → 常驻引擎 → 网络 |
| 引擎存活 | 随 DAW / DLL 生灭 | 常驻服务，独立于客户端 |
| 声道映射 | ❌ 只看得到一个 DAW 的通道，无全局视图 | ✅ 引擎有全局音源 + 总线，可路由 |
| DSP | ❌ 无总线可挂 | ✅ 混音总线上挂 EQ / 限幅 |
| 与第三方设备互通 | 靠流本身合规 | 同左，且能被第三方路由软件纳入矩阵 |

> **结论**：声道映射与 DSP 天然要求「常驻引擎 + 全局总线」，形态 A 撑不起，故本项目主线锁定形态 B。当前代码里的 ASIO 自包含直连是早期抄近路的产物，后续退化为「引擎的一个可选客户端」，不再是主数据通路。

### 1.3 与第三方（顶力等）互通的边界 —— 重要

**真正保证互通的是 AES67 标准本身，不是去适配顶力的私有软件。**

1. **流合规是第一优先级**：SDP 会话描述、RTP 格式（L24 / 48kHz / ptime=1ms）、SSRC / 时间戳时钟域。这直接决定第三方硬件能否订阅到你的流。
2. **发现机制要对齐**：先确认对方环境用 **SAP** 还是 **Ravenna / mDNS** 还是 **NMOS IS-04/05**。只发 SAP、对方跑 mDNS 就发现不了你。这是「能否在对方软件里看到你」的前置条件，比声道映射更靠前。
3. **通道命名要可读**：SDP 的 `i=`、通道 label 与对方约定或人类可读，路由软件才能把你的通道排进矩阵。
4. **不逆向兼容其私有配置格式**：那是对方卖硬件的护城河，适配它既没必要也踩坑。兼容开放标准即可，做到标准合规就天然互操作。

## 2. 系统架构（形态 B 目标态）

```
┌──────────────────────────────────────────────────────────┐
│                   Qt 配置面板 (用户态)                     │
│         状态监控 · 设置 · 声道路由矩阵 · DSP 调参          │
└───────────────────────────┬──────────────────────────────┘
                            │ 命名管道 \\.\pipe\AES67Engine
                            │ (文本命令 + 路由/DSP JSON)
┌───────────────────────────▼──────────────────────────────┐
│              AES67 音频引擎 (用户态 · 常驻服务)            │
│  ┌────────┐┌────────┐┌──────────┐┌──────┐┌───────────┐   │
│  │混音总线││路由表  ││jitter buf││ ASRC ││ DSP 处理链│   │
│  └────────┘└────────┘└──────────┘└──────┘└───────────┘   │
│  ┌────────────┐┌───────────┐┌─────────────┐             │
│  │RTP 收发线程││PTPv2 从时钟││SAP/SDP 通告 │             │
│  └────────────┘└───────────┘└─────────────┘             │
└──────┬───────────────────────────────────┬──────────────┘
       │ 音源输入                            │ RTP/UDP
       │ (阶段①WASAPI回环 / 阶段②驱动共享内存)  │
┌──────▼──────────┐              网络 (PTP 时钟同步) ──► 顶力等 AES67 设备
│  虚拟声卡驱动    │                            ▲
│  (WDM/PortCls)  │◄───────────────────────────┘ 收流 → 虚拟麦克风端点
│  Speaker + Mic  │
└─────────────────┘
```

**核心工程原则**（沿用原始设计，形态 B 下依旧成立）：

- **常驻引擎是地基**：所有智能（路由 / 混音 / RTP / PTP / ASRC / SAP / DSP）集中在用户态引擎；崩了可重启、可脱离驱动单测、可跨平台复用。
- **面板只做展示 / 控制**：通过命名管道与引擎通信，不碰音频。面板崩溃不影响音频流。
- **内核薄**：驱动只负责「制造声卡端点 + 搬字节」，零业务逻辑，绝不在内核跑网络栈。
- **音频线程实时**：无锁、无系统调用、无动态内存；跨线程共享全用 `std::atomic` + Memory Barrier。

### 2.1 音源输入的两个阶段（关键设计取舍）

引擎「拿到整机音源」有两条实现路径，本项目**分阶段落地**：

- **阶段①（先做）WASAPI 回环**：引擎用 WASAPI loopback 抓系统混音。即插即用、零驱动改动，先让形态 B 的路由 / 混音 / 面板整条链路跑起来。
- **阶段②（后做，可选终局）驱动共享内存**：驱动把音频 memcpy 进共享内存，引擎经 IOCTL 读走，不经过 Windows 音频栈，延迟更低、真正「薄内核 + 厚引擎」。这是原始设计的 M2/M3，工程量大、需 WDK，后置。

> 两阶段对上层（路由 / DSP / 面板）透明——上层只面对「引擎的输入总线」，不关心音源是 WASAPI 还是共享内存来的。这让阶段②可以随时替换而不影响已完成的路由 / DSP 功能。

## 3. 技术选型

| # | 模块 | 语言 | 工具 / 库 | 说明 |

## 4. 里程碑主干（M1–M9，形态 B 重定义）

原始 M1–M9 是「双模虚拟声卡」的分解。形态 B 下重新定义其含义与完成状态（详细现状见 `CODE_REVIEW.md`）：

```
M1 驱动骨架(制造声卡端点)  ← 依赖：WDK + Scream 源码
M4 常驻引擎框架            ← 依赖：无   ★形态B地基
   ├─ M5 RTP 发送通路      ← 依赖：M4
   ├─ M6 RTP 接收通路      ← 依赖：M4
   ├─ M7 PTPv2 + ASRC     ← 依赖：M5/M6（跨机才关键）
   └─ M9 面板 + 声道路由    ← 依赖：M4   ★形态B主线，进行中
M2 IOCTL 通道 / M3 共享内存 ← 依赖：M1（阶段②，后置/可选）
M8 ASIO DLL               ← 降级为可选客户端，非主线
```

| 里程碑 | 形态 B 下的定义 | 状态 |
|---|---|---|
| M1 驱动骨架 | 让系统出现虚拟声卡端点（供 WASAPI 抓） | 🟡 Scream 换皮，标识 / 残留未清 |
| M4 常驻引擎框架 | 用户态引擎主控（启停 / 状态机 / 线程） | ✅ 全项目最扎实 |
| M5 RTP 发送 | 独立线程封包发流 | ✅ 能自洽发送（但 SAP 地址 bug 待修） |
| M6 RTP 接收 | recvfrom + jitter buffer | ✅ 能收 |
| M7 PTP + ASRC | 卡尔曼 PTP + libsamplerate 防漂移 | 🟡 PTP 用 EMA；**ASRC 缺失**（跨机才需要） |
| M9 面板 + 路由 | 常驻引擎 + 命名管道 + Qt 面板 + 声道矩阵 | 🟢 **进行中**：pipe + Qt 骨架已成型，路由矩阵未做 |
| M2 IOCTL | 驱动↔引擎控制信道 | ⚪ 阶段②，未动工 |
| M3 共享内存 | 驱动 memcpy 音频给引擎 | ⚪ 阶段②，`g_SharedBuffer` 是死缓冲 |
| M8 ASIO | 降级为可选客户端 | ⚠️ 现为自包含直连，非主线 |

> **形态 B 主干 = M1 → M4 → M9（+ M5/M6）**，跨机场景再补 M7。阶段② M2/M3 是终局优化。ASIO（M8）不在主线上。

## 5. 执行计划（P0 → P8）

### 5.1 协作方式（已确认）

| 项 | 约定 |
|---|---|
| 代码归属 | dwygg/V_AES67，直接在此仓库开发 |
| 协作模式 | **feature 分支 + PR**：每阶段推 `feature/formb-Pxx` 分支，Windows 拉取编译验证，通过后合入 `main` |
| 开发环境 | macOS 侧写代码（用户态 C++/Qt 全可写），**无法本机编译** |
| 验证环境 | Windows + VS2022 + Qt6（+ 后期 WDK）；每次 push 都是**能独立编译验证的完整单元** |
| 验收手段 | 首选：本机自发自收 / 第二台机 + **Wireshark 抓包**验合规；进阶：真实 AES67 设备（顶力）订阅验互通 |

**协作循环**：① mac 写代码推 `feature/formb-Pxx` → ② Windows `git checkout` 该分支编译 → ③ 反馈报错/抓包/现象 → ④ 我改再推，循环到「完成判据」全绿 → ⑤ 合 main 打 tag，进下一阶段。

> ⚠️ 铁律：**每次 push 必须能编过**。我无法本机编译，一次推入多个未验证改动会让往返成本爆炸。宁可拆小、多推几次。

### 5.3 阶段详情

#### P0 — 环境自检（清地基，非开发）
先确认 Windows 侧能把**现有代码**编过，排除「环境问题」与「我的改动问题」混在一起。

| 步骤 | 动作 | 判据 |
|---|---|---|
| 0.1 | `python build.py engine` | 生成 engine，能启动、监听 pipe |
| 0.2 | `python build.py asio` | 生成 `AES67_ASIO.dll` |
| 0.3 | 装 Qt6 + CMake 编 `panel/` | 生成面板 exe，能启动 |
| 0.4 | 记录 VS / Qt / WDK 版本 | 我据此对齐 build 脚本 |

**产出**：一份「Windows 环境 & 首次编译结果」反馈（哪步过、哪步报什么错）。

#### 🟢 P1 — 修 M9 死锁 + pipe 生命周期　`feature/formb-P1-m9-fix`

| # | 问题（详见 CODE_REVIEW §3） | 改动 |
|---|---|---|
| M9-1 | STOP 命令在 pipe 线程里调 `Stop()`，`Stop()` 又 join pipe 线程 → 自锁卡 3s | handler 不直接调 Stop；改为置信号 / 投递到主线程执行 |
| M9-2 | pipe 绑在音频 `Start()`（`aes67_engine.cpp:158`）→ 未 Start 时面板永远连不上 | 把 `m_pipeServer.Start()` 移到 `Initialize()`；`Stop()` 不停 pipe（去掉 `:223`）；pipe 只在 Shutdown 停 |
| M9-3 | SET 只改内存值不重建 socket → 改地址不生效 | SET 后安全重启网络线程，或回报「重启生效」 |
| M9-4 | 面板 Apply 漏发 source port（`m_sourcePort` 从未被读） | 面板补发；引擎 SET 解析使用 |

**完成判据**：① 引擎一启动（未 Start 音频）面板即可连上显示状态；② 面板 START/STOP 正确启停、**不卡顿**；③ 改地址 → Apply → 抓包确认发流目标随之改变。

#### 🟢 P2 — 修 SAP 地址 + 锁死音频格式（让流能被真订阅）⭐　`feature/formb-P2-sap-format`

这是「能不能跟顶力 / 其它 AES67 设备互通」的头号拦路虎。

| # | 问题 | 改动 |
|---|---|---|
| SAP-1 | `aes67_engine.cpp:178` 调 SAP `Start()` 只传 3 个参数，`mcastAddr` 走默认值 `239.255.255.255`（SAP 组地址），SDP `c=IN IP4` 因此通告成 SAP 地址而非真正 RTP 目标 → 接收方订阅不到 | 显式传入真实 RTP 组播目标地址 |
| FMT-1 | `audio_config.h` 默认虽是 48k/24bit/2ch，但 `periodUs=100000`（=100ms，与注释「10ms」不符），AES67 需 **ptime=1ms（48 样本）** | 锁死 L24 / 48kHz / 2ch / ptime=1ms；驱动格式声明也收敛到此档 |
| SDP-1 | SDP 需符合 SMPTE ST 2110-30 / AES67：`m=audio ... RTP/AVP 97`、`a=rtpmap:97 L24/48000/2`、`a=ptime:1`、`a=ts-refclk` | 按标准补齐 SDP 字段；通道命名可读 |

**完成判据**：① Wireshark 抓 SAP：`c=` 是真实 RTP 组播地址，`rtpmap/ptime` 符合 L24/48k/1ms；② 抓 RTP：PT=97、每包 48 样本、时间戳步进 48、1000 pkt/s；③（有设备时）顶力 / 第二台能**发现并订阅**到流、听到声音。

> **P2 末尾 = 第一个关键验收点（MS-2）**：发出的流被真设备订阅到，形态 B 的对外契约立住。

#### 🔵 P3 — 路由契约（JSON）+ 引擎读表　`feature/formb-P3-routing-schema`
定义 `routing.json` schema（`源通道 → 目标流/通道 + 增益 + 静音 + 流描述`）；引擎启动读表，pipe 增加 `GET_ROUTING` / `SET_ROUTING`。此阶段**手改 JSON 即可测通路由逻辑，不依赖界面**。
**判据**：改 JSON → 重启 → 路由 / 流数量 / 增益按表生效（抓包确认多流）。

#### 🔵 P4 — 多源汇聚（常驻引擎混音总线）　`feature/formb-P4-mixer-bus`
引擎内建混音总线：多路输入（先用 WASAPI 回环，阶段②换驱动）按 P3 路由表分发到目标流；总线是 DSP 的挂载点。
**判据**：两路以上音源能按路由表混合 / 分发到不同 AES67 流。

#### 🔵 P5 — Qt 面板接路由矩阵　`feature/formb-P5-panel-matrix`
面板新增「路由矩阵」tab（源×目标网格、增益条、静音钮），通过 pipe 复用 P3 契约同步；面板只展示 / 编辑，逻辑在引擎。
**判据**：面板改路由 → 引擎实时生效 → 抓包可见。

#### 🔵 P6 — DSP 挂载点（增益/静音/EQ 骨架）　`feature/formb-P6-dsp`
混音总线上留 DSP 处理链插槽：先做增益 / 静音 / 简单 EQ，面板可调参。
**判据**：面板调 DSP → 输出音频 / 抓包频谱可见变化。

#### 🟠 P7 — 内核：收流 → 虚拟麦克风输入端点（WDK，最硬）　`feature/formb-P7-capture-endpoint`
当前只有 render（输出），缺 capture（输入）。要让「网络来的 AES67 音频」成为系统录音设备，DAW 才能录进来。需 WDK 新增 capture pin，你在 Windows 侧深度参与（签名 / 测试模式）。
**判据**：系统「录音设备」出现虚拟麦克风，DAW 能录到网络流。

#### 🟠 P8 — 打通 IOCTL + 共享内存（阶段②终局，可选）　`feature/formb-P8-ioctl-shm`
原始 M2/M3：驱动 memcpy 音频到共享内存、引擎经 IOCTL 读写，替换 WASAPI 回环。**可选**：若 WASAPI 回环够用可长期延后。先修 IOCTL 码不匹配（`0x22E0000` vs `0x222003`）与共享内存死缓冲，或干脆删掉这条死通路避免误导。
**判据**：引擎经 IOCTL + 共享内存与驱动交换音频，WASAPI 回环可摘除。

## 7. 代码质量准则

- 内核驱动：C，不用异常 / 标准库 / 动态内存（alloc once at init）；只 memcpy，不解包。
- 用户态引擎：C++17，RAII，智能指针，`std::vector`/`std::array` 代替裸指针。
- 音频线程：无锁、无系统调用、无未预分配的内存操作。
- 跨线程共享变量全用 `std::atomic`；共享内存用 Memory Barrier。
- 日志分级；音频线程内不写详细日志。
- 每模块独立可测——引擎脱离驱动测、驱动脱离网络测。

## 8. 关键工程问题与应对（跨机场景）

| 问题 | 机制 | 应对 |
|---|---|---|
| 爆音·时钟漂移 | 本地晶振与主时钟差几十 ppm，累积误差 | ASRC 动态微调采样率，调幅 ≤ 0.1%（P7/跨机才需要） |
| 爆音·网络抖动 | WiFi 丢包 / 重排 / 延迟尖峰 | jitter buffer 动态窗口 2-6 包；丢包静音填充不中断 |
| 时钟精度不足 | Windows 软件时间戳 ~1ms，PTP 常要求 <100μs | QPC + PreciseTime 双时钟 + 卡尔曼；音频侧 ASRC 补偿即可 |
| 掉驱动 | buffer 位置 / IOCTL 超时 | 驱动只 memcpy，协议全在用户态；位置偏移 >1ms 重置 |
| 签名 | WHQL $400/年 | 开发用测试签名（testsigning on），发布前申请 |

> 同机自发自收（TX/RX 共时钟域）时，ASRC / 高精度 PTP **暂不需要**；跨机组网（AES67 本职、与顶力互通的场景）时它们变为必需。这是 §5 中把 ASRC / 真 PTP 后置的原因。
